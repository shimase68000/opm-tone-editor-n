# Interface Philosophy

Tone Editor exposes OPM parameters directly.

It does not attempt to simplify or abstract FM synthesis.

---

## Principle

> do not hide complexity; present it clearly

---

## Direct Parameter Exposure

FM synthesis is inherently complex.

Instead of masking this complexity,
the editor presents parameters as they are:

- numeric values
- operator-level control
- direct access to all parameters

---

## Implication

- the UI may appear dense to beginners
- parameters are explicit and unabstracted
- no metaphor-based interface is used

---

## Why

Understanding FM synthesis requires:

- direct manipulation
- repeated experimentation
- listening-based learning

Abstraction often hides the relationships between parameters.

Tone Editor avoids this.

---

## Design Goal

The interface is designed to support:

- precision
- repeatability
- deep understanding

---

## Learning Model

Tone Editor assumes that learning happens through:

- changing parameters
- hearing the result
- building internal intuition

The UI is built to support this loop.

---

## Summary

The interface does not simplify FM synthesis.

It exposes it.

This allows users to understand the sound,
not just manipulate it.

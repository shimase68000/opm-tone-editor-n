# Library-Oriented Editing

Tone Editor is designed around the idea that
tone data is not a final product,
but a reusable asset.

---

## Core Principle

> treat all tones as templates

---

## Tone as Template

All existing tones are considered potential templates.

There is no strict distinction between:

- finished tones
- working tones

Every tone can be reused, modified, and refined
to create new results.

---

## Editing Model

Tone editing is not based on creating data from scratch.

Instead, it is based on:

- selecting existing tones
- modifying parameters
- adapting sounds for new purposes

---

## Library as Workspace

Tone files are not treated as static storage.

They are treated as an active workspace
from which tones can be:

- selected
- compared
- reused

---

## Why This Model

In practice, tone editing is often driven by an existing sound.

Users frequently encounter situations such as:

- "this sound is usable"
- "I want to reduce the attack slightly"
- "I want to adapt this sound for my own use"

Editing typically begins from something that already exists.

---

Creating a tone from scratch is possible,
but even for experienced users, it is often difficult
to reach a desired sound directly.

FM synthesis is not inherently intuitive,
and translating an abstract sound image into parameters
requires significant effort and experience.

---

Because of this, Tone Editor assumes that:

> editing is based on modification of existing tones

rather than creation from an empty state.

---

## Implication

This model encourages:

- reuse over recreation
- exploration through variation
- efficient workflow using existing assets

---

## Relation to Interface

This philosophy directly influences the UI design:

- dual file display
- dual parameter tables
- template slots

These features support selecting and modifying tones
across multiple contexts.

---

## Summary

Tone Editor is designed to facilitate iterative refinement
using a reusable tone library.
